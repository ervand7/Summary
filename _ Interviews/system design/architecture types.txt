1. Serverless
Applications run as cloud functions triggered by events. You pay only for the
compute used.
Pros: no infrastructure management, auto-scaling.
Cons: vendor lock-in, harder debugging and monitoring.

2. Monolithic Architecture
The entire system is built as one single application.
Pros: simple development and deployment.
Cons: hard to scale and modify large systems.

3. Microservices
Application is split into independent services, each responsible for a business
function.
Pros: scalability, flexibility, team independence.
Cons: operational complexity, service communication issues.

4. Event-Driven Architecture
Components communicate through events and asynchronous processing.
Pros: scalable, highly responsive.
Cons: complex debugging and data consistency challenges.

5. Client–Server
Clients send requests, servers process them and return results.
Pros: clear separation of responsibilities.
Cons: server dependency, scaling challenges.

6. Layered (N-tier) Architecture
Application is divided into layers (UI, business logic, data).
Pros: good structure, easier testing and maintenance.
Cons: performance overhead, scaling limitations.

7. SPA (Single-Pge Application)
The UI loads once, and data updates dynamically via APIs.
Pros: fast and responsive user experience.
Cons: SEO challenges, heavier client logic.

8. Microkernel (Plugin Architecture)
A minimal core system extended via plugins.
Pros: flexible and extensible.
Cons: plugin management complexity.

9. SOA (Service-Oriented Architecture)
Large systems built from reusable services, often connected via ESB.
Pros: service reuse, flexibility.
Cons: ESB complexity and potential latency.

10. Clean Architecture
Separates business logic from infrastructure and frameworks.
Pros: high testability and maintainability.
Cons: requires careful design.
