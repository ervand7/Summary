Rate limiting can be applied at different layers.

1. Edge (NGINX, Envoy)
Why:
* Blocks spikes early
* Protects all services

Cons:
* Cannot enforce business-specific limits


2. Service-level
Why:
* Different limits per endpoint or user
* Protects DBs and external APIs

Cons:
* Must be implemented in each service


3. Local (per instance)
Why:
* Protects memory, goroutines, and connection pools
* Fastest protection

Cons:
* Limits scale with the number of instances


4. Global (Redis)
Why:
* Enforces limits across all instances
* Supports user and tenant quotas

Cons:
* Additional latency
* Extra infrastructure dependency


Typical approach:
* Edge → traffic spikes
* Service → business rules
* Local → instance protection
* Redis → global quotas

Interview version:
Apply coarse limits at the edge, protect services and instances locally, and use
Redis only when global limits across all instances are required.
