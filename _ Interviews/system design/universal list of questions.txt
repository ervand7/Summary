## 1. Goal and scope

1. What exactly are we designing?
2. Who are the main users?
3. What are the core use cases?
4. What is out of scope?
5. Should we optimize for MVP or production-grade system?

## 2. Scale

6. How many daily/monthly active users?
7. How many requests per second on average?
8. What is the peak RPS?
9. Read-heavy or write-heavy?
10. Expected data size now and in 1-3 years?
11. Any traffic spikes?

## 3. Functional requirements

12. What operations should the system support?
13. Do we need search/filtering/sorting?
14. Do we need real-time updates?
15. Do users need notifications?
16. Do we need admin/internal tools?
17. Do we need APIs for external clients?

## 4. Non-functional requirements

18. What latency is acceptable?
19. What availability target do we need?
20. What consistency level do we need?
21. Can the system tolerate eventual consistency?
22. What should happen during partial failures?
23. What is more important: consistency or availability?
24. Do we need strong durability?

## 5. Data model

25. What entities do we store?
26. What are the relationships between entities?
27. Which fields are indexed?
28. Do we need history/audit logs?
29. Do we need soft deletes?
30. Do we need versioning?

## 6. Database choice

31. Is SQL or NoSQL more appropriate here?
32. Do we need transactions?
33. Do we need joins and complex queries?
34. Do we need horizontal scaling?
35. Do we need sharding?
36. What is the partition key?
37. What are the most important query patterns?

## 7. Caching

38. What data can be cached?
39. What cache TTL is acceptable?
40. How do we invalidate cache?
41. Do we need Redis/Memcached/CDN?
42. What happens if cache is unavailable?
43. Can stale data be shown?

## 8. APIs

44. REST, gRPC, GraphQL, or WebSocket?
45. What are the main endpoints?
46. Do we need pagination?
47. Do we need idempotency keys?
48. Do we need rate limiting?
49. Do we need authentication and authorization?

## 9. Reliability

50. What failures should we expect?
51. Do we need retries?
52. Do we need exponential backoff and jitter?
53. Do we need circuit breakers?
54. Do we need graceful degradation?
55. Do we need disaster recovery?
56. What is acceptable data loss?

## 10. Async processing

57. Do we need a message queue?
58. Kafka, RabbitMQ, SQS, NATS, or something else?
59. Do we need ordering guarantees?
60. At-least-once or exactly-once processing?
61. How do we handle duplicate events?
62. Do we need DLQ?
63. Do we need background workers?

## 11. Security

64. How do users authenticate?
65. What permissions/roles exist?
66. Do we store sensitive data?
67. Do we need encryption at rest/in transit?
68. Do we need audit logging?
69. Do we need protection against abuse/spam?

## 12. Observability

70. What metrics should we track?
71. What logs do we need?
72. Do we need distributed tracing?
73. What alerts should exist?
74. What are the main SLOs/SLIs?
75. How do we debug production issues?

## 13. Deployment and operations

76. Cloud or on-prem?
77. Kubernetes or simpler deployment?
78. How do we deploy safely?
79. Do we need blue-green/canary releases?
80. How do we handle migrations?
81. How do we roll back?

## 14. Product/business constraints

82. Is cost important?
83. Is time-to-market important?
84. Are there legal/compliance requirements?
85. Are there geographic/data residency constraints?
86. Should the system support multiple regions?
87. Do we need multi-tenancy?
