Основная логика GC реализована в файле: src/runtime/mgc.go

GC реализован с использованием алгоритма Mark and Sweep:

Mark — фаза разметки:
- Реализована с помощью трехцветного алгоритма. Он конкурентный и выполняется
параллельно с основной программой.
  - Все данные в куче представляются в виде графа связанных объектов.
  - Каждый объект может быть окрашен в один из трёх цветов: белый, серый, черный.
  - Обход начинается с корневых объектов (например, глобальных переменных).
  - Объекты, на которые есть ссылки, становятся черными.
  - Серые — это объекты, которые были найдены, но их ссылки еще не обработаны.
  - Белые — это объекты, на которые нет ссылок; они считаются "мертвыми" и будут удалены.

Sweep — фаза очистки:
- Удаляются объекты, оставшиеся белыми после фазы Mark.
- Выполняется конкурентно (без stop-the-world), часто лениво при новых аллокациях.

Дополнительно:
- GC можно вызвать вручную с помощью функции `runtime.GC()`.
- На работу GC по умолчанию выделяется около 25% CPU, используемого приложением.

📌 Этапы фазы Mark:
 - Mark Setup – STW
Очень короткая пауза. Подготовка: все потоки приостанавливаются, чтобы GC мог
безопасно начать фазу Mark (зафиксировать корни: stack, globals).

 - Marking – ✅ Concurrent
Главная часть фазы Mark. GC работает параллельно с вашим кодом.
Потоки продолжают выполнять программу, а GC просматривает граф объектов в фоне.

 - Mark Termination – STW
Завершающая короткая пауза. GC проверяет, не появились ли новые объекты, и завершает разметку.

То есть STW происходит 2 раза.
