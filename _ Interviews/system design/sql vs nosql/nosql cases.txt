Sure.

> I prefer SQL by default because it provides transactions, consistency, and strong
querying capabilities.
>
> I choose NoSQL when I need flexible schemas, very high write throughput, or massive
horizontal scalability.

Examples:

* SQL → banking, payments, orders, because transactions and consistency are critical.

* Product catalogs (NoSQL) → different products have different attributes. A TV, a
laptop, and shoes may have completely different fields, so a flexible schema is
convenient.

* Event tracking (NoSQL) → systems may collect millions of clicks, views, and user
actions per day. The workload is mostly writes, and NoSQL databases often scale
horizontally more easily.

* Social media content (NoSQL) → posts, comments, reactions, media metadata, and
user-generated content evolve frequently. The schema changes often, and the system
needs to handle huge volumes of data and traffic.

A very short interview version:

> SQL is best when consistency and transactions matter. NoSQL is often preferred
when data structure changes frequently or when the system needs to handle massive
volumes of data and writes at scale.
