MVCC (Multi-Version Concurrency Control)

MVCC lets databases (e.g., PostgreSQL) handle concurrent reads/writes without locking.

### 🧠 Core idea

* Store multiple versions of rows instead of overwriting
* Readers see a snapshot (consistent view)
* Writers create new versions

### 🔧 Isolation levels (via MVCC)

* READ COMMITTED → latest committed data each query
* REPEATABLE READ → same snapshot for whole transaction
* SERIALIZABLE → + conflict checks for full consistency

### 📦 Example

* A reads → sees `v1`
* B updates → creates `v2`
* A still sees `v1` (snapshot isolation)

### 📚 Key points

* No dirty reads
* Repeatable reads (stable data in txn)
* Phantom reads possible (except SERIALIZABLE)
* Old versions cleaned by VACUUM

### 💥 Pitfalls

* Long transactions → block cleanup
* Concurrent writes → conflicts / retries
* READ COMMITTED → inconsistent multiple reads

---

## ✅ Summary

* MVCC = versioning instead of locking
* Isolation level = which versions you see
* Great for high concurrency, but watch transactions & conflicts
