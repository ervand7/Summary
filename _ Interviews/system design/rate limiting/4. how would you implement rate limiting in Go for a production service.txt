Production rate limiting usually combines local protection with per-user limits.

1. Middleware / interceptor
* Per-user, API key, or IP limits
* Different limits per endpoint

Why:
* Protects DBs and external services
* Supports business-specific rules


2. Local concurrency limit
* Semaphore or worker pool
* Max concurrent requests per instance

Why:
* Protects memory, goroutines, and connection pools
* Fastest protection


3. Global limiter (Redis)
* Limits shared across all instances

Why:
* Enforces true per-user quotas
* Prevents bypassing limits via multiple pods


Production features:
* 429 Too Many Requests
* Retry-After header
* Metrics for allowed/denied requests
* Configurable limits

Typical flow:
* Request → local limiter → optional Redis limiter → handler

Interview version:
Use middleware/interceptors for per-user limits, local concurrency caps for instance
protection, and Redis only when global limits across all instances are required.
