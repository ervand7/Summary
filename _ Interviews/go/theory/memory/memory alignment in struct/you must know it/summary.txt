### 1️⃣ Why padding exists

Go aligns fields in a struct to match the CPU’s memory alignment rules (for faster access).
If fields are arranged poorly, the compiler inserts padding bytes between them.

Example:

```go
type Bad struct {
    a bool  // 1 byte
    b int64 // 8 bytes
    c bool  // 1 byte
}
```

Layout (approx):

```
a: 1 byte
padding: 7 bytes
b: 8 bytes
c: 1 byte
padding: 7 bytes
= 24 bytes
```

### 2️⃣ Order fields from largest → smallest

To reduce padding:

```go
type Good struct {
    b int64
    a bool
    c bool
}
```

Now:

```
b: 8
a: 1
c: 1
padding: 6
= 16 bytes
```

### 3️⃣ Why it matters

It improves:

 Memory usage (important for large slices/maps of structs)
 CPU cache efficiency
 GC performance

Example:
1 million structs × 8 bytes saved = ~8 MB less memory.

### 4️⃣ How to check struct size

```go
import "unsafe"

fmt.Println(unsafe.Sizeof(MyStruct{}))
```

### 5️⃣ Tools that help

Useful tool:

```
go vet -structtag
```

or external:

```
fieldalignment
```

Example:

```
go install golang.org/x/tools/go/analysis/passes/fieldalignment/cmd/fieldalignment@latest
fieldalignment ./...
```

### 6️⃣ When to care

Optimize when:

 structs are stored in large slices
 used in hot paths
 used in high-load services

Ignore when:

 small number of objects
 readability would suffer.

---

✅ Senior-level takeaway (one sentence):

> Arrange struct fields from largest to smallest to minimize padding, especially for 
structs used in large collections or performance-critical paths.
